## Modes

Claude Code has two primary operating modes: **Plan** and **Edit (implement)**. Knowing when to use each — and which model to pair with each — is how you get the best results on non-trivial tasks.

---

### Plan Mode

In plan mode, Claude reads the codebase and thinks through the approach **without writing any code**. It produces a step-by-step plan, identifies the files it will touch, and surfaces trade-offs and ambiguities before anything changes.

Invoke it with:
```
/plan
```

Or just ask Claude to plan first:
```
Before writing anything, plan how you'd add streaming support to the REPL.
```

**What Claude does in plan mode:**
- Reads relevant files to understand the current structure
- Identifies what needs to change and in what order
- Notes risks, edge cases, and decisions that need your input
- Does NOT edit, write, or run commands

**When to use it:**
- Before any non-trivial feature or refactor
- When you're not sure what the full scope of a change is
- When you want to review the approach and give feedback before code gets written
- When a task touches multiple files or modules

---

### Edit Mode (default)

The default mode. Claude reads, edits, runs commands, and iterates until the task is done. No constraints — full tool access.

This is what Claude does when you just describe a task without asking for a plan first.

---

### Why the Separation Matters

Planning and implementing require different things from the model:

- **Planning** is about understanding, reasoning, and identifying the right approach. It benefits from deeper thinking and broader context awareness.
- **Implementing** is execution — applying a known plan to specific files. It's more mechanical once the plan is clear.

Mixing them means Claude might start writing code before it fully understands the scope, then have to backtrack. Separating them gives you a checkpoint: you review the plan, correct any wrong assumptions, then let Claude execute with confidence.

---

### Using a Better Model for Planning

Because planning is higher-stakes reasoning, you can assign a more capable model to it and use a faster/cheaper model for implementation.

In practice:
- Use **Opus** (most capable) for planning complex features, architecture decisions, or anything where getting the approach wrong is costly
- Use **Sonnet** (default, fast) for implementing the plan — writing code, running tests, fixing lint errors

You can switch models mid-session in Claude Code, or configure the model per task type.

**Example workflow:**

```
# Switch to Opus, plan the feature
/model opus
Plan how to add system-prompt support to the REPL without breaking existing conversation flow.

# Review the plan, then switch to Sonnet for implementation
/model sonnet
Implement the plan.
```

This gives you Opus-quality architecture decisions at the cost of only one planning call, then fast Sonnet execution for the rest.

### Model Reference

| Model | ID | Best for |
|---|---|---|
| Claude Opus 4.6 | `claude-opus-4-6` | Planning, architecture, complex reasoning |
| Claude Sonnet 4.6 | `claude-sonnet-4-6` | Implementation, iteration, day-to-day work |
| Claude Haiku 4.5 | `claude-haiku-4-5-20251001` | Simple edits, fast lookups, high-volume tasks |

### Plan Mode in CLAUDE.md

The workflow in this project's CLAUDE.md encodes this pattern as a rule:

> **Plan first.** Before writing code, outline the steps. Ask for clarification on ambiguous trade-offs.

This ensures Claude doesn't start editing until the approach is agreed on — regardless of which mode you're in.
