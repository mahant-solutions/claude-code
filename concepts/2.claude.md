## CLAUDE.md

`CLAUDE.md` is a plain markdown file that Claude Code reads at the start of every conversation. It is how you give Claude persistent, project-specific instructions without repeating yourself in every prompt.

Think of it as the onboarding doc Claude reads before touching your codebase.

---

### Where It Lives

| Location | Scope |
|---|---|
| `~/.claude/CLAUDE.md` | Global — applies to all projects |
| `<project-root>/.claude/CLAUDE.md` | Project-scoped — applies to this repo only |
| `<subdir>/CLAUDE.md` | Directory-scoped — applies when working inside that subdirectory |

Claude merges all of them. More specific files take precedence.

---

### What Goes In It

A good `CLAUDE.md` answers: _"What does Claude need to know to work in this codebase without asking me basic questions every time?"_

Useful sections:
- **Project overview** — what the app does, what stack it uses
- **Package structure** — where things live, what each module owns
- **Workflow rules** — how to approach tasks (plan first, test after each step, etc.)
- **Coding standards** — style, patterns, conventions that aren't obvious from the code
- **Tooling** — what commands to run for tests, lint, type checks
- **What NOT to do** — explicit guardrails ("never put HTTP logic in `repl.py`")

---

### What Does NOT Go In It

- Code patterns or architecture already visible in the codebase
- Ephemeral task state (use `TodoWrite` for that)
- Secrets or credentials
- Anything already in a spec or README that Claude can just read

---

### Example: This Project

`.claude/CLAUDE.md` in this repo contains:
- A module responsibility map (`client.py` = HTTP only, `conversation.py` = history only)
- The 5-step workflow (plan → implement → test → verify → bump spec version)
- Coding standards: SOLID principles, type annotation rules, error handling conventions
- Testing conventions: file placement, class structure, mocking approach
- A tooling table with every `uv run` command

This means Claude never has to ask "where do tests go?" or "how do I run the linter?" — it already knows.

---

### Syntax Tips

- Use headings to organize sections
- Use tables for quick-reference info (commands, module map)
- Use bold for rules: `**Rule:** Keep each concern in its own module.`
- Keep rules imperative and specific: "Never put cross-module logic in `main.py`" beats "Try to keep things separate"
- Rules that include _why_ are more useful than rules without context

---

### Rule Hierarchy

If a `CLAUDE.md` rule conflicts with Claude's default behavior, the `CLAUDE.md` wins. If a rule is ambiguous, Claude will ask. If you want zero ambiguity, be explicit: "Always X. Never Y."
