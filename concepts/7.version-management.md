## Version Management

When building with a coding agent, the most important question is: **when do you commit?**

The answer maps directly to how you structure your work.

---

### The Phase-Based Mental Model

A coding agent moves fast. Without deliberate checkpoints, you end up with a large undifferentiated blob of changes that's hard to review, hard to roll back, and hard to reason about.

The antidote is the same principle used in this project's `specs/engineering_design_doc.md`:

> **Each phase is self-contained: it adds one vertical slice of functionality, ends with passing tests, and leaves the codebase in a working state.**

This is your commit boundary. **One phase = one commit.**

---

### What "Phase" Means in Practice

A phase is a unit of work that:
- Has a single clear goal ("build the HTTP client", "add the renderer")
- Ends with a green verification gate (tests pass, lint clean, types clean)
- Leaves the codebase working — not half-built
- Can be code-reviewed as a coherent, standalone change

From this project:

| Phase | What it built | Commit message |
|---|---|---|
| 1 | Project skeleton + conversation model | `Add project skeleton and conversation data layer` |
| 2 | Ollama HTTP client with streaming | `Add OllamaClient with streaming support` |
| 3 | Rich terminal renderer | `Add Renderer for terminal output` |
| 4 | REPL loop + CLI entry point | `Wire REPL, slash commands, and CLI entry point` |
| 5 | Error handling + edge cases | `Harden error handling and signal management` |

Each of these is a reviewable unit. If Phase 3 introduced a bug, you can `git revert` it without touching Phase 2.

---

### The Rule

```
commit after every phase, never in the middle of one
```

**Never commit a broken codebase.** If the tests aren't green, the phase isn't done. If the phase isn't done, don't commit.

**Never batch multiple phases into one commit.** It collapses context, makes rollback painful, and defeats the purpose of phasing.

---

### Verification Gate = Commit Gate

The engineering design doc defines an explicit gate for every phase:

```bash
uv run pytest                  # tests pass
uv run ruff check .            # lint clean
uv run ruff format --check .   # formatting clean
uv run pyrefly check .         # types clean
```

If any of these fail, the phase is not done. Do not commit. Fix it first.

This gate is the contract between you and the agent: the codebase must be healthy at every checkpoint.

---

### Why This Matters More With an Agent

With a human developer, commits are a habitual checkpoint. With a coding agent, work happens in bursts — an agent can implement an entire phase in one response. The risk isn't forgetting to commit; it's letting the agent continue into the next phase before you've reviewed and committed the last one.

**Establish the checkpoint yourself.** After each phase:
1. Review the diff
2. Run the verification gate
3. Commit
4. Then ask the agent to proceed to the next phase

This keeps you in control of what's in the codebase and gives you a clean rollback point at every phase boundary.

---

### Branching

For multi-phase features, work on a feature branch:

```bash
git checkout -b feature/streaming-support
# Phase 1 commit
# Phase 2 commit
# Phase 3 commit
git merge main  # or open a PR
```

Each phase commit on the branch is reviewable. The PR diff is the full feature. Either way, the history is readable.
