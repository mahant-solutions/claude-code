## MCPs (Model Context Protocol)

MCP is a standard protocol that lets Claude connect to external tools and data sources — beyond what it can do with just files and bash.

Without MCPs, Claude only knows what's in its training data and the current codebase. MCPs give it live access to external systems: documentation, databases, design files, APIs, browser state, and more.

---

### How It Works

An MCP server exposes a set of **tools** that Claude can call, just like it calls `Read` or `Bash`. The difference is the tools are provided by an external process, not built into Claude Code.

```
Claude Code  →  MCP Client  →  MCP Server  →  External System
                (protocol)      (your tool)     (docs, DB, API...)
```

You configure MCP servers in `~/.claude/settings.json` or the project's `.mcp.json`. Once registered, their tools appear alongside Claude's built-in tools.

---

### Example: Context7 (used in this project)

Context7 is an MCP server that fetches live, version-accurate documentation for any library.

**Why it exists:** Claude's training data has a cutoff. If you're using a newer version of `httpx`, `typer`, or `rich`, the API may have changed. Context7 fetches the actual current docs instead of Claude guessing from stale training data.

**Tools it provides:**
- `resolve-library-id` — find the library's ID in the Context7 index
- `query-docs` — fetch documentation for a specific topic from that library

**How it's used in this project:**

The global rule at `~/.claude/rules/context7.md` instructs Claude to always use Context7 when working with libraries:

1. Call `resolve-library-id` with the library name
2. Pick the best match (prefer version-specific IDs)
3. Call `query-docs` with the library ID and the question
4. Answer using the fetched docs, citing the version

The CLAUDE.md also reinforces this: _"Use context7 MCP for up-to-date documentation on `httpx`, `rich`, `typer`, or any other library."_

### Other Common MCPs

| MCP | What it gives Claude access to |
|---|---|
| **Supabase** | Query and mutate your Supabase database, manage tables, run SQL, inspect schema |
| **Figma** | Read design files, extract component specs, inspect layers and styles |
| **GitHub** | Read issues, PRs, repo contents, create PRs, comment — beyond what `gh` CLI does |
| **Postgres / SQLite** | Direct database access — query, explain, inspect schema |
| **Browserbase / Playwright** | Control a real browser — navigate, click, screenshot, scrape |
| **Slack** | Read channels, post messages, search message history |
| **Linear** | Read and create issues, update project status |
| **Filesystem (extended)** | Access paths outside the current working directory |

### Where MCPs Are Configured

**Global** — applies to all projects:
```json
// ~/.claude/settings.json
{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp"]
    }
  }
}
```

**Project-scoped** — committed to the repo:
```json
// .mcp.json (project root)
{
  "mcpServers": {
    "supabase": {
      "command": "npx",
      "args": ["-y", "@supabase/mcp-server-supabase@latest"],
      "env": {
        "SUPABASE_URL": "${SUPABASE_URL}",
        "SUPABASE_SERVICE_ROLE_KEY": "${SUPABASE_SERVICE_ROLE_KEY}"
      }
    }
  }
}
```

### MCPs vs Built-in Tools

| | Built-in tools | MCP tools |
|---|---|---|
| Examples | `Read`, `Edit`, `Bash`, `Grep` | `query-docs`, `query-database`, `get-figma-file` |
| Source | Claude Code itself | External server process |
| Scope | Local filesystem & shell | External systems |
| Config needed | No | Yes — register in settings |
